# Base Agent Behaviors

All agents inherit these behaviors. Read this FIRST before your agent-specific instructions.

---

## 1. Pre-Flight: Load Learned Patterns (TIERED)

**BEFORE investigating or planning**, load accumulated knowledge and apply it.

- **Preferred — MCP** (vault-metrics, `_v1` versioned aliases only):
  `failure_patterns_v1()` (failure patterns w/ frequency) +
  `agent_metrics_v1(period="30d")` (success rates by complexity/stack).
- **Fallback — files** (MCP unavailable): `cat .claude/memory/patterns-critical.md`
  (always; ~50 lines, covers 89% of failures) and, for COMPLEX issues,
  `patterns-full.md` (generated by `/learn`; skip if absent).
- **Pull coding-memory on demand**: when you hit a *specific* question about a
  subsystem/pattern/prior decision you're touching, query the semantic store with
  the problem at hand: `~/agents/bin/coding-memory query "<your question>"` (then
  Read the source file of any hit for the body). This is precise, problem-specific
  recall — use it rather than relying only on the `{CODING_MEMORY_BLOCK}` the
  orchestrator pushed. Fail-open: if the store is unreachable, just proceed.

---

## 2. Pre-Flight: Check Similar Past Work

Find similar past artifacts (`grep -l "KEYWORD" .agents/outputs/*.md 2>/dev/null | head -3`);
if found, read them to learn what worked and what caused issues.

---

## 3. Efficiency Rules

- **Reference, don't re-quote.** Use `See backend/accounts/services.py:45-67` instead of pasting code.
- **Single source of truth.** Acceptance criteria defined ONCE (in MAP-PLAN or PLAN). Other agents reference, don't duplicate.

### Target Lengths

Each agent's `target_lines` / `max_lines` live in its own frontmatter. Before
submitting, run `wc -l < .agents/outputs/$ARTIFACT_NAME`. Over `max_lines`:
**STOP and compress** before submitting; between target and max: submit with a note.

**Compression order**: (1) replace code quotes with line refs, (2) reference acceptance criteria instead of re-stating, (3) consolidate duplicate sections, (4) drop appendices / "Future Enhancements", (5) report exceptions only (failures, not successes).

---

## 4. Artifact Naming

Pattern: `{agent}-{issue}-{mmddyy}.md` written to `.agents/outputs/`. Set `ISSUE_NUMBER`, `RUN_DATE=$(date +%m%d%y)` at start of run. Full spec: `~/.claude/rules/orchestrate-workflow.md`.

---

## 4.5 Persisting Your Output (CRITICAL — canonical)

You have the **Write** tool. You MUST Write your final output to your
frontmatter `output:` path BEFORE emitting `AGENT_RETURN`, substituting `{issue}`
(the issue number) and `{mmddyy}` (today in MMDDYY). Skipping this stalls the
workflow — the orchestrator and the Artifact-Validation gates read the file on
disk, not your chat response. This is the single source of truth; agent files
reference it rather than restating it.

## 6. Constraint Enforcement

Git/branch/PR constraints live in `~/.claude/rules/git-workflow.md` (auto-loaded);

## 10. Root Cause Classification (Canonical Enum)

When recording failures, use ONLY these root cause codes:

| Code | Description |
|------|-------------|
| `ENUM_VALUE` | Used enum NAME instead of VALUE |
| `COMPONENT_API` | Wrong props/hook usage |
| `MULTI_MODEL` | Forgot model relationship |
| `API_MISMATCH` | Frontend/backend contract violation |
| `ACCESS_CONTROL` | Missing/wrong permission check |
| `MISSING_TEST` | Untested code path |
| `SQLITE_COMPAT` | PostgreSQL-only feature used |
| `STRUCTURE_VIOLATION` | Violated rules.md constraints |
| `SCOPE_CREEP` | Beyond issue scope |
| `VERIFICATION_GAP` | Assumptions not verified by reading code |
| `OTHER` | Document specifics in `details` field |

## 11. Canonical JSONL Schemas (metrics + failures)

The orchestrator writes both records (see `commands/orchestrate.md` Step 4, the
canonical recording site). Agents only populate the data via artifact
frontmatter — they do NOT append JSONL themselves.

**`metrics.jsonl`** required fields: `issue`, `date`, `status`
(PASS|BLOCKED), `complexity` (TRIVIAL|SIMPLE|COMPLEX), `stack`
(backend|frontend|fullstack), `agents_run` (list), `agent_versions` (map),
`root_cause` (null unless BLOCKED), `blocking_agent` (null unless BLOCKED),
`duration_minutes`. Optional: `recovery_attempts`, `notes`.

**`failures.jsonl`** required fields: `issue`, `date`, `agent`, `root_cause`
(a §10 code), `details`, `fix`, `prevention`, `files` (list). Optional:
`severity`, `recovery_minutes`.

## 13. Outcome Recording (Orchestrator-Driven)

PROVE does NOT write outcome records directly — the orchestrator does
(`commands/orchestrate.md` Step 4, calling `state_manager.record_metrics` /
`record_failure`). PROVE's job is to specify the data via artifact frontmatter:
`status`, `complexity`, `stack`, plus `root_cause` (§10 code) and
`blocking_agent` when BLOCKED. The §11 schemas define what those helpers write.
Why: an `echo >> file` step at the end of PROVE's prompt proved unreliable
(#104); a deterministic Python call closes the gap.

---

## 14. Agent Versioning

All agents carry `version: X.Y` in frontmatter — minor (1.0→1.1) for pattern/wording additions, major (1.0→2.0) for restructure/new sections/workflow changes. Record current versions in each metrics record's `agent_versions` field (§11) so `/metrics` can correlate success rates with versions.

## 15. Escalation Policy

When BLOCKED (waiting on user, ambiguous requirement, missing access):

| Duration | Action |
|----------|--------|
| < 2 min | Wait for user input |
| 2-5 min | Proceed with safest assumption, tag with `[ASSUMED]` in code/artifact |
| > 5 min | STOP, report assumption + what needs human verification |

**Hard escalation (STOP immediately, no assumptions)**: complexity misclassified, constraint violation required, security-sensitive ambiguous scope, or same approach failed twice with same root cause (the **two-failure rule** → STOP, escalate to user or `/codex:rescue`).

PROVE flags any `[ASSUMED]` tags for human review — document each assumption (what / why / safest alternative).

---

## 16. Anti-Pattern Self-Check

Before each phase, name any you're exhibiting and course-correct.

| Anti-Pattern | Signal → Correction |
|-------------|----------------------|
| **Kitchen Sink** | Fixing unrelated issues → revert scope; file extras separately |
| **Correcting Over and Over** | Same fix applied 3+ times → approach is wrong, try a different strategy |
| **Infinite Exploration** | Read 15+ files, no plan → stop, write a hypothesis, verify targeted files only |
| **Trust Then Verify Gap** | Assuming code works because it looks right → run it (`ruff`, `pytest`, `npm run build`) |
| **Architectural Astronautics** | Abstractions for hypothetical future needs → delete; build for today's requirement |
| **Flip-Flop** | Implement A → revert → implement B → revert → … → stop, validate with data first |

---

## 17. Runbook Check

On any error, check `cat .claude/memory/runbooks.md 2>/dev/null` BEFORE investigating from scratch. If the error matches a runbook entry, apply the documented fix directly — don't re-diagnose known problems.

---

## 18. Failure Context Awareness

A `## Prior Failure` block in your prompt is **highest-priority context** (a
prior PATCH failed on this exact issue): read its root-cause + prevention, apply
the prevention BEFORE starting, verify the failure point is addressed, and note
in the artifact `Prior failure (ROOT_CAUSE) addressed by: [action]`.

---

## 19. Swarm-Aware Behavior

When spawned as a scoped sub-task (e.g., PATCH-backend, PROVE-backend): touch
only files in your scope; CONTRACT is the boundary for parallel fullstack PATCH;
name scoped artifacts `{agent}-{scope}-{issue}-{mmddyy}.md`; document
discovered out-of-scope needs under "Cross-Scope Dependencies" instead of making
the change; flag any file appearing in both scopes immediately.
